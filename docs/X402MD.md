# X402 Agent Reserve Standard

This document captures the BuddyPie implementation shape for x402-funded,
per-agent reserves.

## Why this model

BuddyPie runs long-lived coding workspaces, not single exact-price API calls.
That makes a reserve model the safest fit today:

- x402 funds the user's balance.
- BuddyPie allocates part of that balance to a specific agent preset.
- The preset spends from that reserve across many sandbox actions.
- Internal leases prevent concurrent overspend.
- Actual usage is debited from the reserve and recorded in an immutable ledger.

This follows the current x402 guidance:

- `exact` requires strict equality.
- `up-to` is still a roadmap direction.
- Usage-based charging should sit behind an internal ledger or credit system.

## Current BuddyPie scope

Implemented now:

- user funding account in USD, backed by USDC on Base Sepolia
- per-user, per-agent reserve allocation
- reserve holds for sandbox launch and post-launch billable actions
- immutable usage events and ledger entries
- dashboard UI for funding and reserve refill
- sandbox detail UI for billed usage history

Current limitation:

- top-ups require users to connect a Base Sepolia wallet and sign a USDC transfer
- wallet verification currently checks exact transfer amount and recipient only

The reserve system stays framework-agnostic. The implemented seller route and
the manual path both finalize funding into the same internal mutation.

## Reserve key

BuddyPie uses:

```text
(user_id, agent_preset_id, currency=USD, network=base-sepolia, asset=USDC, environment=prod)
```

Today the "agent" dimension is the BuddyPie preset:

- `general-engineer`
- `frontend-builder`
- `docs-writer`

Adding a new preset automatically makes it eligible for the same allowance
system.

## Data model

Convex tables:

- `billingAccounts`
- `fundingTransactions`
- `agentReserves`
- `reserveLeases`
- `usageEvents`
- `ledgerEntries`

Sandbox records also store:

- `agentReserveId`
- `launchLeaseId`
- `billedUsdCents`
- `lastBilledAt`

## Pricing model in this repo

BuddyPie currently prices the observable platform actions it controls:

- `sandbox_launch`
- `preview_boot`
- `ssh_access`
- `web_terminal`

Price version:

```text
buddy_x402_2026_03_16_v1
```

This is intentionally honest about current observability. BuddyPie does not yet
receive provider-accurate OpenCode token-cost callbacks, so the ledger meters
known sandbox lifecycle events first. When OpenCode usage hooks become
available, the same reserve and ledger model can ingest token-based usage
events.

## Runtime flow

### 1. Fund wallet

Implemented x402 seller flow:

1. Client hits `POST /v1/wallet/topups`
2. x402 seller middleware returns `402 Payment Required`
3. Buyer wallet signs and retries
4. Seller verifies payment and calls the existing funding finalization path
5. BuddyPie credits `billingAccounts.unallocatedUsdCents`

Current dashboard fallback flow:

- dashboard calls a manual settled-top-up server action
- the app server posts to a Convex HTTP action with the user's Convex auth token
- the HTTP action calls the internal funding-finalization mutation
- the mutation records a `fundingTransactions` row and credits unallocated funds
- expired active leases are released by a Convex cron sweeper

### 2. Allocate or refill agent reserve

The dashboard moves money from the funding account into a preset reserve:

- debit `funding_unallocated`
- credit `reserve_available`

### 3. Launch sandbox

Before provisioning starts:

- create a short-lived launch lease
- move `reserve_available -> reserve_held`

On success:

- capture the lease into a `usageEvents` row
- move held funds into `revenue`
- write ledger entries

On failure:

- release the hold back to `reserve_available`

### 4. Bill post-launch actions

Preview boot, SSH access, and the Daytona web terminal also use the reserve:

- create a short-lived lease
- execute the action
- capture on success
- release on failure or if the action was already running

## Standard integration point for real x402

The seller entrypoint is implemented as:

- `POST /v1/wallet/topups`

This endpoint requires an authenticated Convex user, returns `402` with
`PAYMENT-REQUIRED` when payment headers are missing, verifies and settles via
an x402 facilitator, and finalizes crediting through:

- `recordFundingTopup`

Wire the x402 seller middleware to the same logical mutation used by the manual
top-up flow:

- `recordFundingTopup`

`recordFundingTopup` is intentionally internal-only. Trusted entrypoints should
be:

- a Convex HTTP action
- or a server/admin client path outside the browser-facing public Convex API

Expected inputs from the x402 layer:

- payment reference / transaction identifier
- exact funded USD amount
- Base Sepolia network
- USDC asset
- idempotency key based on the x402 payment id

Recommended idempotency format:

```text
funding:<x402_payment_id>
```
